# KServe Project Overview

KServe is a Kubernetes-native platform for serving machine learning models. The Go codebase implements custom controllers using controller-runtime to manage inference services, serving runtimes, and related resources on Kubernetes.

The project lives under `github.com/kserve/kserve`. Core controller logic is in `pkg/controller/`, API types in `pkg/apis/`, and webhook handling in `pkg/webhook/`.

## Building and Testing

### Prerequisites

- Go (version pinned in `go.mod`)
- `make`

### Building

```bash
make manager       # build controller manager binary
make all           # run tests, then build manager, agent, and router binaries
```

### Testing

```bash
make test          # full Go test suite (downloads envtest assets automatically)
make test-qpext    # qpext tests
```

Run tests for a specific package:

```bash
make setup-envtest
KUBEBUILDER_ASSETS="$(./bin/setup-envtest use $(go list -m -f '{{ .Version }}' k8s.io/api | awk -F'[v.]' '{printf "1.%d", $3}') -p path)" go test ./pkg/controller/v1beta1/...
```

Controller tests are a mix of unit tests (often with `fake.NewClientBuilder`) and envtest integration suites. Envtest is used across multiple controller areas, including llmisvc.

### Linting and Formatting

```bash
make fmt           # gofmt
make vet           # go vet
make go-lint       # golangci-lint
make precommit     # all pre-commit checks at once
```

### Code Generation

```bash
make generate                # codegen, OpenAPI/Python client generation, Helm chart READMEs
make manifests               # CRDs, RBAC, webhook configs
make generate-chart-manifests  # Helm chart manifests from kustomize overlays
```

Several artifacts are autogenerated - do not edit them by hand:
- Helm chart `README.md` files are generated by `helm-docs` from `values.yaml` annotations
- Helm chart templates under `charts/*/templates/` are generated from kustomize overlays via `generate-chart-manifests`
- Common Helm helpers (`_utils.tpl`, `_common.tpl`, `_resources.tpl`) are synced from `charts/_common/` to individual charts
- CRD YAML files are copied from `config/crd/full/` into `charts/kserve-crd/templates/`

`make precommit` runs all of these in the right order.

## Development Conventions

- API types live in `pkg/apis/serving/` with versioned subdirectories (`v1alpha1`, `v1alpha2`, `v1beta1`).
- Controllers live in `pkg/controller/` with versioned subdirectories.
- Tests live next to the code they test. Pure logic/unit tests often use fakes, while controller integration suites typically use envtest.
- CRDs are generated from Go types using controller-gen markers.
- Run `make precommit` before submitting changes to catch formatting, linting, and generation drift.

## Development Workflow

Follow a test-first, iterative approach:

1. **Analyze** - understand the task, identify affected controllers/types, search for reusable patterns
2. **Test first** - write the test before implementation code. Use envtest for controller reconciliation, standard Go tests for pure logic
3. **Implement** - minimal, focused changes. Follow existing project patterns
4. **Verify** - run selective tests, then `make precommit` before committing
5. **Review** - check for correctness, idiomatic Go, and controller-runtime best practices

## Pull Requests

Use the template in `.github/PULL_REQUEST_TEMPLATE.md` when opening PRs. Every PR must include:

- **What this PR does / why we need it** - clear description of the change and motivation
- **Which issue(s) this PR fixes** - link issues with `Fixes #<number>` format
- **Feature/Issue validation/testing** - describe how you tested the change, include relevant logs
- **Checklist** - confirm unit/e2e tests, comments for complex code, and docs updates
- **Release note** - write a release note or `NONE` if not user-facing

Run `make precommit` before opening a PR.

## Controller-Runtime Patterns

### Reconcile Loop

- Reconcile must be **idempotent** - same input run N times produces the same result
- Use `context.Context` everywhere; avoid `context.Background()` inside reconcile
- Handle `NotFound` as success for deleted objects
- Use `Patch` with `MergeFrom` for updates to reduce conflicts
- Return errors for reconciliation failures (controller-runtime handles backoff). Use `Requeue: true` only for async work in progress, `RequeueAfter` only for wall-clock delays
- Short-circuit when no-op - already-compliant objects should reconcile fast with no API calls

### Status and Conditions

- `spec` holds desired state (user-owned), `status` holds observed state (controller-owned)
- Always include `observedGeneration` in conditions
- Use conditions (not phase fields) with CamelCase types: `Ready`, `Succeeded`, etc.
- Update status only when there is a real change

### Watches and Caching

- Use predicates to drop irrelevant events early
- Use `Owns()` and targeted watches over broad `Watches()` where possible
- Add field indexers to avoid expensive list+filter patterns
- Use `APIReader` only when cache staleness causes real correctness issues
- The cached client has no read-your-write consistency - writes hit the API server but are not instantly visible in cache

### Testing with envtest

- envtest runs API server and etcd but NOT built-in controllers (Deployment, ReplicaSet, etc.) or garbage collection
- Each test creates its own namespace for isolation
- Clean up resources with `defer` immediately after creation
- Use `Eventually`/`Consistently` for async assertions, never `time.Sleep`
- Simulate external controller behavior (status updates) when needed using helper functions
- Use `retry.RetryOnConflict` when updating resources the controller is also reconciling
- Prefer table-driven tests for pure logic, envtest integration tests for controller wiring
- Use `_test` packages (e.g., `package llmisvc_test`) for new tests to avoid coupling test code with implementation details. Some older controllers still use same-package tests - don't change existing tests, but write new ones in `_test` packages
